# Release Notes — 19 maj 2026

## DuckLake Access Manager · v14–v19

Sex versioner under maj. Fokus på tre saker: automatisk rensning av nycklar när access återkallas, en gräns för hur många nycklar en student kan ha, och möjligheten för admin att ladda upp Parquet-filer direkt i webbgränssnittet.

---

## v14 — Automatisk nyckelrensning vid grant-återkallelse

Tidigare hade systemet ett gap: om admin tog bort en students tillgång till ett dataset levde studentens S3-nyckel och Postgres-konto kvar. Studenten kunde alltså fortsätta läsa data trots att access hade återkallats.

Nu raderas alla berörda nycklar automatiskt när:

| Admin-åtgärd | Vad raderas automatiskt |
|---|---|
| Tar bort ett `user`-grant | Studentens nyckel för det datasetet |
| Tar bort ett `group`-grant | Alla gruppmedlemmars nycklar för det datasetet |
| Tar bort en student ur en grupp | Studentens nycklar för alla datasets gruppen har access till |

Rensningen är *best-effort*: om Garage eller Postgres är temporärt otillgänglig loggas felet men grant-återkallelsen genomförs ändå. Nyckeln kanske dröjer någon minut, men access-raden försvinner omedelbart — studenten ser inte längre datasetet i Browse.

---

## v15 — Maximal en aktiv nyckel per dataset och student

Studenter är nu begränsade till **en aktiv nyckel per dataset**. Om man försöker generera en ny nyckel för ett dataset man redan har en nyckel till visas ett tydligt meddelande direkt på kortet:

> *You already have an active key for this dataset. Delete it first.*

Ingen alert-dialog längre — felmeddelandet visas inline under "Generate Keys"-knappen i gult.

Admins berörs inte av begränsningen och kan generera hur många nycklar som helst.

**Varför denna begränsning?** Utan den kunde studenter samla på sig obegränsat många nycklar (och Postgres-konton) för samma dataset, vilket försvårar administration och städning.

## v16 — Bättre formulärvalidering för dataset- och gruppnamn

Två buggar fixades i formulären för att skapa datasets och grupper:

**Frontend:** Fälten för bucket-namn och gruppnamn saniterar nu inmatningen automatiskt medan man skriver. Versaler görs om till gemener, mellanslag ersätts med bindestreck, och ogiltiga tecken filtreras bort. En formatledtråd visas under varje fält.

**Backend:** Valideringsregexen för gruppnamn tillät felaktigt namn som slutar med bindestreck (t.ex. `my-group-`). Fixat — sista tecknet måste nu vara en bokstav eller siffra, precis som för dataset-namn.

---

## v17 — Admin kan ladda upp Parquet-filer via webbgränssnittet

Admin har nu en ny flik i gränssnittet: **Upload**.

Flödet är enkelt:
1. Välj vilket dataset filen ska tillhöra
2. Ange ett tabellnamn (används som mapp-prefix i S3)
3. Välj en `.parquet`-fil från din dator
4. Klicka Upload

Filen hamnar i datasetets Garage-bucket på sökvägen `data/<tabellnamn>/<filnamn>`. Systemet validerar att det faktiskt är en giltig Parquet-fil (magic bytes `PAR1`) innan uppladdningen påbörjas.

Alla uppladdade filer visas i en lista per dataset med filstorlek och uppladdare. Filer kan raderas från listan — de tas då bort från Garage S3 också.

**Teknisk not:** Uppladdningen sker via ett tillfälligt S3-nyckelpar som skapas och raderas automatiskt runt varje uppladdning. Inga permanenta admin-credentials behöver konfigureras. Maximal filstorlek är 500 MB.

## v18 — Nyckelrensning vid grupps radering

En bugg i v14: om admin raderade en hel grupp rensades aldrig nycklar för gruppens medlemmar. Bara `removeMember()`-flödet hade rensning — inte `deleteGroup()`.

Nu körs samma rensningslogik även vid grupps radering: alla aktiverade nycklar för alla gruppmedlemmar på alla datasets gruppen hade access till raderas innan gruppen tas bort.

---

## v19 — Nyckelrensning vid @everyone-återkallelse och dataset-radering

Ytterligare två gap i rensningslogiken som nu är åtgärdade:

**@everyone-grant:** När admin stängde av `@everyone`-access på ett dataset (dvs. ändrade det från publikt till privat) raderades inte befintliga nycklar. Användare som genererat nycklar medan datasetet var publikt behöll fortfarande fungerande S3-åtkomst. Nu raderas alla nycklar för datasetet när `@everyone`-grantet återkallas.

**Dataset-radering:** När admin raderade ett dataset togs bucket och Postgres-databas bort, men nycklar i Garage och `key_user_mapping`-rader i systemdatabasen lämnades kvar som skräp. Nu rensas alla nycklar för datasetet automatiskt innan bucket raderas.

## Sammanfattning

| Version | Fokus |
|---------|-------|
| v14 | Automatisk nyckelrensning vid grant-återkallelse och grupptillhörighet |
| v15 | Max en nyckel per student och dataset |
| v16 | Formulärvalidering — auto-sanitering och regex-fix |
| v17 | Admin kan ladda upp Parquet-filer via webbgränssnittet |
| v18 | Bugg: nyckelrensning saknades vid grupps radering |
| v19 | Bugg: nyckelrensning saknades vid @everyone-återkallelse och dataset-radering |

Med v19 är alla kända gap i nyckelrensningslogiken täckta. Alla vägar som tar bort access — grant-återkallelse, grupptillhörighet, grupps radering, dataset-radering — rensas nu konsekvent.
