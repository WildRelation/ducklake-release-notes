# Release Notes — 10 maj 2026

## DuckLake Access Manager · v10–v13

Den 10 maj gjordes de hittills största förändringarna i systemet. Fyra versioner deployades under kvällen. Fokus låg på att gå från ett system som hanterade enskilda buckets och studenter, till ett riktigt kurshanteringsverktyg där lärare kan administrera hela klasser och projekt strukturerat.

---

## Bakgrund — hur det såg ut innan

Tidigare bestod systemet av:
- En delad Postgres-databas (`ducklake`) för all metadata
- Enskilda S3-buckets utan koppling till varandra
- Ett enkelt grant-system: "student X får tillgång till bucket Y"

Det fungerade för en-till-en-scenarion men skalade dåligt. En lärare som ville ge 30 studenter tillgång till en bucket behövde lägga till varje student manuellt.

---

## Vad är nytt?

### 1. Dataset — ett arbetsområde för en kurs eller ett projekt

Nu introduceras **Dataset** som det centrala konceptet i systemet. Namnet kan vara lite missvisande — ett dataset här är inte en enskild datafil eller tabell, utan snarare ett **arbetsområde eller namnrum** som kan innehålla hur många tabeller och parquet-filer som helst.

Konkret: ett dataset är en S3-bucket och en Postgres-katalog-databas som hör ihop:

```
Dataset: "kurs2026"
├── S3-bucket: kurs2026/          ← här lagras datat (parquet-filer etc.)
│   ├── students.parquet
│   ├── results.parquet
│   └── logs.parquet
└── Postgres: dl_kurs2026         ← DuckLake-katalog (håller koll på tabeller och filer)
```

DuckDB kopplar ihop båda i ett kommando:

```sql
ATTACH 'ducklake:postgres:dbname=dl_kurs2026' AS my_ducklake (
    DATA_PATH 's3://kurs2026/'
);
```

Postgres fungerar alltså som en **katalog** — vilka tabeller finns, var ligger filerna — och S3 är där datat faktiskt bor. Studenter jobbar mot `my_ducklake` och ser tabeller precis som i en vanlig databas, utan att behöva tänka på var filerna ligger.

Varje dataset har även **metadata**: titel, beskrivning, ägare, synlighet (publik/privat).

När ett dataset skapas skapas S3-bucketen och Postgres-databasen atomärt. Om något steg misslyckas rullar systemet tillbaka det som hann skapas. När ett dataset raderas tas allt bort i ett steg — bucket, databas och alla grants.

**Postgres används nu på två nivåer:**

| Nivå | Databas | Syfte |
|------|---------|-------|
| System | `ducklake` | Metadata — nycklar, användare, datasets, grupper, grants |
| Per dataset | `dl_<dataset-namn>` | Studentens egna tabeller och kataloginformation |

Varje student som genererar en nyckel till dataset `kurs2026` får ett Postgres-konto scopat **enbart** till databasen `dl_kurs2026`. De kan inte se eller ansluta till andra dataset — full isolering.

Befintliga buckets från innan uppdateringen hanteras automatiskt: vid uppstart registrerar systemet alla Garage-buckets som saknar en dataset-rad och skapar tillhörande Postgres-databaser, utan manuell migration.

### 2. Grupper — hantera hela klasser på en gång

Lärare kan nu skapa **grupper** — namngivna samlingar av studenter (e-postadresser). Grupper kan tilldelas tillgång till ett dataset i ett steg, och alla nuvarande och framtida medlemmar i gruppen ärver automatiskt den tillgången.

**Tre typer av grants stöds nu:**

| Typ | Vad det innebär |
|-----|-----------------|
| `user` | En specifik student via e-post (som tidigare) |
| `group` | Alla medlemmar i en namngiven grupp |
| `everyone` | Alla inloggade användare i systemet |

Gamla grants från det tidigare systemet (`student_grants`) migreras automatiskt till det nya formatet vid uppstart — ingen dataförlust, ingen manuell insats.

Typiskt flöde för en lärare:
1. Skapa gruppen `databaskurs-ht26`
2. Importera alla studenter
3. Ge gruppen tillgång till kursens dataset — klart

---

### 3. Bulk-import av studenter

För att fylla en grupp utan att lägga till studenter en i taget finns nu **bulk-import**. En lista med e-postadresser klistras in direkt i gränssnittet — separerade med radbrytning, komma eller semikolon (t.ex. kopierat från ett kursregister eller LMS).

Systemet svarar med ett kvitto:

```json
POST /api/groups/databaskurs-ht26/members/bulk
→ { "added": 28, "skipped": 2, "invalid": 0 }
```

- **added** — nya unika adresser som lades till
- **skipped** — adresser som redan var medlemmar
- **invalid** — adresser som inte ser ut som giltiga e-postadresser

---

### 4. `.env`-fil vid nyckelgenerering

När en student genererar sina inloggningsuppgifter kan de nu ladda ner en färdig **`.env`-fil** direkt från gränssnittet. Filen innehåller alla anslutningsuppgifter med rätt variabelnamn för de vanligaste Python-biblioteken:

```dotenv
# PostgreSQL — kompatibel med psycopg2, SQLAlchemy, libpq
PGHOST=postgres.example.com
PGPORT=5432
PGDATABASE=dl_kurs2026
PGUSER=u_abc123
PGPASSWORD=...

# S3 — kompatibel med boto3, s3fs, DuckDB
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
AWS_DEFAULT_REGION=garage
AWS_ENDPOINT_URL=http://s3.example.com
S3_BUCKET=kurs2026
```

Filen heter `.env.<dataset-namn>` och kan läggas direkt i projektroten. Inga manuella konfigurationssteg. Vid sidan av `.env`-filen genereras även ett färdigt DuckDB-script som ansluter till rätt dataset-databas.

---

### 5. Robustare nyckelgenerering med rollback

Nyckelgenerering skapar resurser i två separata system (Garage och Postgres). Om ett steg misslyckas städar systemet nu upp efter sig:

- **S3-nyckel skapas, Postgres-steget misslyckas** → S3-nyckeln raderas
- **Båda skapas, mappning till användare misslyckas** → både S3-nyckeln och Postgres-användaren raderas

Resultatet är att inga halvfärdiga resurser lämnas kvar i systemet, oavsett vad som går fel.

## Sammanfattning

| Område | Innan 10 maj | Efter 10 maj |
|--------|-------------|--------------|
| Resurshantering | Bucket + delad Postgres | Dataset = bucket + isolerad PG-databas |
| Accesshantering | En student i taget | Grupper, bulk-import, everyone-grant |
| Databaseisolering | Alla studenter i samma DB | En DB per dataset, scopade konton |
| Komma igång | Manuell konfiguration | Ladda ner `.env`, kör igång |
| Felhantering vid nyckelgenerering | Möjliga orphans | Automatisk rollback |

Systemet tar med detta ett tydligt steg från ett verktyg för att administrera enskilda användare, till en plattform som kan hantera hela kurser och projekt från dag ett.
