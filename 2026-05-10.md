# Release Notes — 10 maj 2026

## DuckLake Access Manager

Den 10 maj gjordes de hittills största förändringarna i systemet. Fokus låg på att göra det möjligt för lärare att hantera hela klasser och dataset på ett strukturerat sätt, samt att göra det enklare för studenter att komma igång.

---

## Vad är nytt?

### Datasets — ett samlat begrepp för dataresurser

Nu introduceras **Dataset** som ett enhetligt koncept: ett dataset består av en S3-bucket och en dedikerad Postgres-databas som hör ihop. Varje dataset har ett namn, en ägare och kan delas med specifika användare eller grupper.

Postgres används nu på två nivåer i systemet:

- **Systemdatabas (`ducklake`)** — fanns redan sedan tidigare. Lagrar metadata om nycklar, användare, datasets och grupper.
- **Per-dataset-databaser (nytt)** — när ett dataset skapas skapas också automatiskt en helt egen Postgres-databas för just det datasetet (t.ex. `ds_kurs2026`). Studenter som får tillgång till datasetet får ett eget Postgres-konto scopat till den databasen och inget annat — full isolering mellan kurser och projekt.

Det gör det enklare att tänka på resurser ur ett kurs- eller projektperspektiv snarare än på infrastrukturnivå.

### Grupper — hantera hela klasser på en gång

Lärare kan nu skapa **grupper** — namngivna samlingar av studenter (e-postadresser). Istället för att lägga till varje student manuellt till ett dataset kan man ge en hel grupp tillgång på ett steg.

Typiskt användningsfall: skapa gruppen `databaskurs-ht26`, lägg till alla studenter, och ge sedan gruppen tillgång till kursens dataset.

```
POST /api/groups/databaskurs-ht26/members/bulk
→ { added: 28, skipped: 2, invalid: 0 }
```

### Bulk-import av studenter

För att fylla en grupp snabbt finns nu en **bulk-importfunktion**. Admins kan klistra in en lista med e-postadresser (kopierat från t.ex. ett kursregister) direkt i gränssnittet — separerade med radbrytning, komma eller semikolon. Systemet lägger till alla giltiga adresser och rapporterar hur många som lades till, hoppades över eller var ogiltiga.

### `.env`-fil vid nyckelgenerering

När en student genererar sina inloggningsuppgifter kan de nu ladda ner en färdig **`.env`-fil** med rätt variabelnamn för de vanligaste Python-biblioteken:

- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / `AWS_ENDPOINT_URL` / `S3_BUCKET` — för `boto3` och `s3fs`
- `PGHOST` / `PGPORT` / `PGDATABASE` / `PGUSER` / `PGPASSWORD` — för `psycopg2` och `SQLAlchemy`

Filen heter `.env.<dataset-namn>` och kan läggas direkt i projektet. Inga manuella konfigurationssteg.

### Robustare nyckelgenerering

Om något tekniskt går fel mitt under nyckelgenereringen (t.ex. att en del lyckas men en annan misslyckas) städar systemet nu upp efter sig automatiskt. Inga halvfärdiga resurser lämnas kvar.

```java
// Om S3-nyckeln skapas men Postgres-steget misslyckas:
garageAdminClient.deleteKey(s3KeyId);  // städa upp
```

## Varför är detta viktigt?

Innan dessa förändringar var systemet byggt för att hantera enskilda studenter och enskilda buckets. Med dagens release tar systemet ett steg mot att vara ett riktigt **kurshanteringsverktyg** — lärare kan sätta upp en hel kurs med ett dataset och en grupp, importera alla studenter på en gång, och studenter kan komma igång direkt utan att konfigurera något manuellt.
